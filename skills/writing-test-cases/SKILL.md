# Writing Test Cases

## 概述

把 PRD、验收标准或 feature spec，收敛成 `测试计划 + 可执行测试用例`。

这份 skill 不负责重写需求，也不负责实现自动化脚本。它只负责把“完成定义”翻译成“QA 如何验证”。

## 何时使用

适用于：
- 已有 PRD、功能需求或验收标准，需要继续产出测试内容
- 需要把 `FR / AC` 变成覆盖清晰、优先级明确的测试设计
- 需要压缩测试文档，避免写成冗长的步骤堆砌

不要用于：
- 需求仍模糊、关键业务决策未定，应先回到 `reviewing-product-requirements`
- 自动化脚本编写、框架选型、技术测试实现

## 核心规则

- 先审输入质量，再写测试；输入不完整时降级为 `测试草案`
- 先做测试建模，再写步骤；不要把 PRD 逐条翻成 case
- 默认只覆盖关键路径、失败态、边界、权限、回归风险
- 每条 case 都必须能追踪到 `FR` 或 `AC`
- 输出默认精简：优先可执行和可追踪，不做穷举矩阵

## 工作流

1. 输入审查
- 检查是否至少具备：目标/范围、功能需求、验收标准、约束或依赖
- 若存在会改变测试范围的待决问题，结果标记为 `测试草案`

2. 测试建模
- 先识别：主路径、失败态、边界值、角色/权限、状态切换、迁移/兼容、回归影响
- 多角色场景按角色拆分，不要混在一条 case 里

3. 生成测试计划
- 写清测试目标、范围、优先级、环境前提、风险与不测项
- 明确 `P0` 必测范围，防止所有 case 都变成最高优先级

4. 生成详细用例
- 一条 case 只验证一个主行为
- 步骤短、预期可观察、标签清晰
- 同类 case 合并表达，避免数量膨胀

## 默认输出结构

默认只保留以下内容；不适用的章节直接省略。

1. `测试状态`
- `完成态测试集` 或 `测试草案`
- 若是草案，先列 `输入缺口 / 待确认问题`

2. `测试目标`

3. `测试范围`
- `In Scope`
- `Out of Scope`

4. `测试策略`
- 主路径
- 失败态
- 边界值
- 权限 / 角色
- 回归
- 迁移 / 兼容（适用时）

5. `优先级`
- `P0`
- `P1`
- `P2`（只保留必要代表项）

6. `环境与前置条件`
- 只写非显而易见的账号、权限、数据、依赖

7. `覆盖追踪`
- `AC-1 -> TC-001, TC-002`
- `FR-3 -> TC-006`

8. `详细测试用例`
- `TC-001`
- 标题
- 对应需求 / 验收标准
- 优先级
- 前置条件
- 步骤
- 预期结果
- 标签

## 长度控制

- 先给“最小可上线测试集”，不要默认穷举所有组合
- 小功能默认 6-12 条 case；中等功能默认 12-20 条；更大范围先分主题，不一次性写满
- `P0` 只覆盖上线阻断风险；不要把所有 case 都标成 `P0`
- 重复路径优先参数化或合并，不重复抄整段步骤
- 预期结果只写判定点，不写调试动作和实现猜测
- 若用户只要测试计划，不展开详细 case

## 评审检查

至少检查这些点：
- 输入是否足够支撑测试设计
- 核心验收标准是否都有覆盖
- 是否只写了主路径而漏掉失败态、边界或回归
- 不同角色/权限是否被错误混写
- 迁移、兼容、旧行为共存是否在适用时被覆盖
- 预期结果是否可观察、可判定

## 完成定义

满足以下条件，才可输出 `完成态测试集`：
- 输入需求足够清晰，不存在关键待决问题
- 核心验收标准已映射到测试覆盖
- 高风险路径、失败态、边界、权限和回归影响已体现到计划或 case
- 用例优先级清晰，`P0` 覆盖最小上线风险

否则输出 `测试草案 + 输入缺口/待确认问题`。

## 与 PRD 的交接

- 优先使用 `reviewing-product-requirements` 产出的完成态 PRD
- 若 PRD 中存在 `Open Questions`，默认只能输出 `测试草案`
- 若 PRD 已写 `发布 / 迁移说明`，测试计划必须覆盖切换、共存和回退验证
