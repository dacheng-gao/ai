# Reviewing Product Requirements

## 概述

把零散、模糊、实现导向的需求，收敛成一份短而可执行的 PRD。

目标不是“写得像 PRD”，而是让产品、研发、测试能基于同一份约定继续推进。

## 何时使用

适用于：
- 粗糙想法、GitHub issue、会议纪要、评论串，需要收敛成正式需求
- 已有 PRD，但过长、过散、偏实现，需要压缩和纠偏
- 需要把“想做什么”与“为什么做、做到什么算完成”分开

不要用于：
- 已经进入技术设计、实现方案或任务拆解
- 需求仍缺关键业务决策，且用户尚未确认默认假设

## 核心规则

- 先定义问题，再评价方案；避免把用户提出的做法直接当需求
- 默认做范围压缩，优先保住 `v1` 的最小可交付
- 只写影响用户体验、范围、验收、风险的内容，不写技术实现细节
- 输入不完整时输出 `PRD 草案`，不要伪装成完成态
- 输出文档默认精简：每节 1-3 个 bullets，非适用章节直接省略

## 工作流

1. 规范化输入
- 用 3-5 行写清：目标、范围、不做、关键假设、当前输入类型
- 输入若是 issue/comment，先区分：用户问题、提议方案、实现提示

2. 做最小挑战
- 是否存在 XY 问题：用户在要“办法”，还是在定义“问题”
- 标题是否过窄、误导或实现导向
- `v1` 是否混入迁移、回写、策略系统、企业边角场景
- 是否缺少用户、非目标、约束、成功标准、失败态、依赖
- 会影响范围或上线方式的业务决策，必须提升为 `Open Questions`

3. 生成精简 PRD
- 先写可观察行为，再写范围与约束
- 每个核心目标都应落到 `FR`、验收标准或成功指标
- 无法验证的需求，不算完成

4. 收口
- 删除背景铺垫、实现推测、重复解释、空泛路线图
- 确保文档能直接贴进 issue、planning thread 或 docs

## 默认输出结构

默认只保留以下内容；不适用的章节直接省略。

### 必选

1. `PRD 状态`
- `完成态 PRD` 或 `PRD 草案`
- 若是草案，先列 `Open Questions`

2. `背景 / 问题`
- 当前痛点是什么
- 为什么值得做

3. `目标`
- 2-4 个明确目标

4. `非目标`
- 明确 `v1` 不做什么

5. `目标用户 / 关键场景`
- 只写主要用户和主场景

6. `范围`
- `In Scope`
- `Out of Scope`

7. `功能需求`
- `FR-1`, `FR-2`, ...
- 写系统应表现出的行为，不写实现步骤

8. `验收标准`
- 只写可观察结果
- QA 应能直接据此设计测试

9. `风险 / 待定决策`
- 产品风险、范围风险、依赖风险
- 真正影响范围或排期的问题才进入这里

### 选填

仅在确实影响决策或上线时加入：
- `约束与依赖`
- `成功指标`
- `拆期`
- `发布 / 迁移说明`

## 长度控制

- 默认目标是“短文档可决策”，不是“全量说明书”
- 单节超过 3 个 bullets 时，优先合并或删减
- 背景不要超过 3 句
- `功能需求` 默认控制在 5-9 条；超过时先检查是否范围膨胀
- `验收标准` 只覆盖关键可观察结果，不展开成测试步骤
- 没有迁移、指标、拆期需求时，不补这些章节凑完整

## 评审检查

至少检查这些点：
- 问题是否独立于方案存在
- `v1` 范围是否被悄悄放大
- 关键约束、依赖、权限是否入文
- `Goal -> FR -> Acceptance` 是否能追踪
- 验收标准是否可测试
- 待决业务问题是否被如实暴露，而不是藏进“假设”

## 完成定义

满足以下条件，才可输出 `完成态 PRD`：
- 问题、目标、非目标、范围已明确
- 关键约束和依赖已写清，不依赖口头补充
- 核心目标可追踪到 `FR` 或验收标准
- 不存在会改变范围、优先级或上线方式的关键未决问题

否则输出 `PRD 草案 + Open Questions`。

## 下游衔接

- PRD 进入完成态后，如需继续产出测试计划和测试用例，使用 `writing-test-cases`
- 若当前仍是 `PRD 草案`，先补齐待决问题，再进入测试设计
