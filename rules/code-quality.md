# 代码质量规范

代码变更交付前必须通过以下门禁：
- 必查 4 项：正确性、安全、性能、可维护性
- 验证按适用性执行（见第 5 节）
- review 自检必须执行最小优化回路（见第 6 节）

纯文档/问答任务可跳过编译与测试。

## 1) 正确性

- 需求行为完整，边界条件可解释
- 错误处理清晰，不吞异常
- 不引入明显回归（接口、时序、输出格式）
- 共享可变状态有明确同步机制
- 资源（连接、句柄、流）及时释放

## 2) 安全

- 外部输入先校验（类型/长度/格式/范围）
- 禁止 SQL/命令拼接，使用参数化或安全 API
- 禁止硬编码密钥、token、凭证
- 受保护资源必须做鉴权与归属检查
- 敏感数据最小暴露：日志脱敏，避免泄漏个人信息与凭证

## 3) 性能

- 热路径避免 O(n²) 及以上复杂度
- 禁止循环内重复 I/O、分配或查询
- 禁止不必要的大对象拷贝或序列化

## 4) 可维护性

- 单一职责：函数/模块职责清晰
- 命名可读：变量用名词，函数用动词
- 复杂逻辑补充“为什么”的注释
- DRY：2+ 处相同/高度相似代码必须消除重复（提取函数/模块/共享组件）
- YAGNI：禁止推测性功能、未请求抽象、不可能场景的错误处理
- 复杂度检验：若资深工程师会说“改得太多/太复杂”，则继续简化
- 手术式改动：单次变更只解决当前目标，避免顺手重构无关区域
- Minimal：优先复用现有实现，非必要不新增抽象/依赖/配置
- Dead code 清理：触及范围内的未使用导入、变量、函数、分支必须清理
- 注释规范：只写“为什么/约束/陷阱”，禁止复述代码；过时注释必须同步删除
- 鲁棒性：关键路径显式处理边界与失败策略，禁止依赖脆弱隐式假设

## 5) 验证

按优先级执行全部适用验证（禁止只挑最少项）：

1. Typecheck / Build：不过则后续验证无意义
2. Lint：风格与潜在错误检查
3. 测试：能跑必须跑；不能跑要说明原因、残余风险与替代方式
4. 手动验证：无法自动化时补充

- 报告必须包含验证证据（完整命令 + 输出摘要 + 受限项）
- 禁止未执行验证却声称“已验证”

## 6) Review 自检优化回路

每轮实现结束后必须执行一次 review 自检，并自动修复触及范围内的低成本问题：

1. Reviewer 输出两类项：`阻断项`、`低成本优化项`（死代码、冗余注释、可读性噪音、局部低效实现）。
2. Implementer 必须先修复 `阻断项`，并默认修复触及范围内全部 `低成本优化项`。
3. 若优化会引入高风险行为变化，必须显式标注为 `defer` 并说明风险与后续计划。
4. 每轮修复后按影响范围重新执行第 5 节验证（Typecheck/Build → Lint → Test）。
5. 同一问题最多迭代 3 轮；仍未收敛时必须在交付中披露残余风险。

## 红旗

### 立即阻断（出现即停止当前工作并修复）

- 安全：SQL/命令拼接、动态执行不受控代码（eval）、未校验输入直接入库/执行
- 安全：敏感信息写入日志或硬编码到代码
- 正确性：关键错误被忽略或静默失败

### 优先修复（在通用退出标准第 4 步质量门禁中处理）

- 跨文件重复代码块
- 单次使用场景的过度抽象
