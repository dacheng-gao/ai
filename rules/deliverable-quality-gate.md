# 交付物质量门禁（Deliverable Quality Gate）

本规则是 Codex / Claude 等 AI Agents 的统一交付质量门禁，适用于代码、脚本、文档，以及其他可交付产物的质量判定与汇报。

## 1. 适用范围与目标

- 适用范围：代码、脚本、文档；其他产物默认先适用本规则的通用门禁，再按最接近的专项门禁执行
- 目标：确保交付物达到 `industry / enterprise / production-ready` 的最低质量要求，而不是仅“看起来完成”
- 使用方式：在声称“已完成”“已通过”“可交付”之前，必须按本规则检查全部适用项
- 适用性原则：不适用的条目必须显式标记 `N/A` 并说明原因，禁止跳过不写

## 2. 规范性术语与判定模型

### 2.1 规范性术语

- 术语语义采用 `RFC 2119 / RFC 8174` 的常见规范性解释
- `MUST`：强制要求；未满足即不能判定为通过
- `SHOULD`：强烈建议；未满足时必须说明理由与风险
- `MAY`：可选增强项；按场景采用

### 2.2 判定结果

- `PASS`
  - 所有适用 `MUST` 已满足
  - 无未披露阻断风险
  - 验证证据完整且与结论一致
- `CONDITIONAL PASS`
  - 所有适用 `MUST` 已满足
  - 存在非阻断缺口，但风险、影响、补救计划已披露
  - 不得包含安全或正确性阻断项
- `FAIL`
  - 任一适用 `MUST` 未满足
  - 存在高风险未处理或未披露
  - 无法提供最小验证证据却声称完成

### 2.3 Waiver（例外）

- `waiver` 仅用于临时接受非理想状态，不得伪装成 `PASS`
- `waiver` 不得绕过阻断级安全问题、关键正确性缺陷、虚假验证、已知重大误导
- `waiver` 必须记录：
  - `Item`
  - `Reason`
  - `Risk`
  - `Why acceptable now`
  - `Follow-up action`
  - `Expiry`

## 3. 交付汇报最低格式

AI agent 交付时必须至少输出：

- `Artifact Type`
- `Scope`
- `Result: PASS | CONDITIONAL PASS | FAIL`
- `Evidence`
- `Risks`
- `Waiver`（如无则写 `none`）
- `Done / Partial / Skipped`

禁止在没有新鲜验证证据的情况下使用“已完成”“已通过”“可上线”等成功性表述。

## 4. 通用门禁

以下条目适用于所有交付物。

### 4.1 目标达成（Goal Completion）

- `MUST` 完成用户/业务目标与显式验收标准
- `MUST` 明确列出未完成项、降级项、跳过项
- `MUST NOT` 以“代码更优雅”“文档更完整”等替代真实目标完成
- `SHOULD` 在汇报中区分功能完成、质量提升、附带优化

### 4.2 正确性与事实性（Correctness & Factuality）

- `MUST` 保证行为、步骤、结论、示例、命令与实际产物一致
- `MUST` 处理适用的边界条件、失败路径或限制条件
- `MUST NOT` 吞掉关键错误、伪造结果、隐瞒不确定性
- `SHOULD` 对回归风险、兼容性影响、假设前提做出说明

### 4.3 安全与隐私（Security & Privacy）

- `MUST` 对外部输入、外部命令、外部数据来源执行最小必要校验
- `MUST` 防止常见高危问题：注入、越权、敏感信息泄漏、不安全默认值、危险动态执行
- `MUST` 禁止硬编码密钥、令牌、凭证、内部敏感配置
- `MUST` 对日志、文档、示例中的敏感信息做最小暴露与脱敏
- `SHOULD` 遵循最小权限、最小暴露、secure-by-default
- `SHOULD` 对新增依赖、脚本来源、第三方片段、供应链来源做最小可信度审查，优先可信来源、可追溯版本与已知安全基线

### 4.4 可靠性与可运维性（Reliability & Operability）

- `MUST` 明确失败时的行为：报错、退出、回滚、重试、告警或人工介入方式
- `MUST` 释放或关闭适用资源，避免遗留脏状态
- `SHOULD` 让关键路径具备最小可排障性，如错误上下文、可追踪日志、诊断信息
- `SHOULD` 对高风险操作说明前置条件、影响范围与恢复方式

### 4.5 可维护性与可读性（Maintainability & Readability）

- `MUST` 保持手术式改动，只解决当前目标，不顺手扩大范围
- `MUST` 保持结构清晰、命名可读、职责边界明确
- `MUST` 仅添加必要注释，重点解释 `why / constraint / trap`
- `MUST NOT` 引入过度抽象、推测性功能、无请求依赖、重复实现扩散
- `SHOULD` 优先复用已有模式和结构，降低认知负担

### 4.6 验证与证据（Verification & Evidence）

- `MUST` 先识别可证明结论的验证命令或检查方式，再声称结果
- `MUST` 对所有适用项执行验证；不能执行时必须说明原因、风险与替代证据
- `MUST` 在汇报中给出命令、检查方式或回读证据摘要
- `MUST NOT` 用“应该可以”“理论上通过”“改动很小无需验证”等表述替代证据

### 4.7 可追溯性与范围控制（Traceability & Scope Control）

- `MUST` 说明本次交付对应的目标、范围与非目标
- `MUST` 对关键结论提供可追溯证据：文件、命令、链接、需求来源
- `SHOULD` 说明哪些条目是新增、修改、删除
- `SHOULD` 对重要文档与代码改动建立最小关联，如需求 → 实现 → 验证

## 5. 专项门禁

### 5.1 代码（Code）

#### 5.1.1 业务目标与行为

- `MUST` 先完成业务目标与验收行为，再讨论优雅性
- `MUST` 明确输入、输出、状态变化、副作用
- `MUST` 评估接口、格式、时序、共享状态变更带来的回归风险

#### 5.1.2 安全

- `MUST` 校验外部输入的类型、长度、格式、范围或权限
- `MUST` 使用参数化、安全 API 或框架原生机制，避免 SQL/命令/模板/路径拼接漏洞
- `MUST` 对鉴权、授权、归属检查、敏感数据访问做显式控制
- `MUST` 避免引入 `OWASP Top 10`、`OWASP ASVS`、`CWE Top 25` 覆盖的常见高危问题

#### 5.1.3 架构与模块化

- `MUST` 保持合理分层与模块边界，避免跨层耦合与职责漂移
- `MUST` 让函数/类/模块职责单一且名称可读
- `SHOULD` 让依赖方向稳定、接口边界清晰、实现可替换
- `SHOULD` 优先局部重构，不为一次性场景引入公共抽象

#### 5.1.4 可维护性与可读性

- `MUST` 清理触及范围内的死代码、未使用导入、显著冗余逻辑
- `MUST` 仅在复杂逻辑、约束、陷阱处添加必要注释
- `MUST NOT` 用注释掩盖糟糕命名或混乱结构
- `SHOULD` 保持函数短小、控制流可读、异常处理显式

#### 5.1.5 可靠性与运行特性

- `MUST` 对资源释放、超时、重试、幂等性、并发一致性按适用性处理
- `SHOULD` 为关键失败路径保留最小可诊断信息
- `SHOULD` 避免热点路径上的明显低效实现，如不必要的 O(n²)、循环内重复 I/O、过量拷贝

#### 5.1.6 验证

- `MUST` 依次执行适用的 `Typecheck / Build → Lint / Static Analysis → Test → Manual Verification`
- `MUST` 对行为变更提供足够的自动化或手动验证
- `SHOULD` 对高风险修复提供回归验证或复现证据

### 5.2 脚本（Scripts）

#### 5.2.1 目标与执行语义

- `MUST` 明确脚本用途、输入、输出、退出码、前置条件
- `MUST` 对副作用进行说明，如文件修改、网络访问、权限变更、外部系统写入
- `SHOULD` 在可行时保持幂等；若非幂等，必须显式警告

#### 5.2.2 执行安全

- `MUST` 校验参数、路径、环境变量、依赖命令、权限
- `MUST` 对删除、覆盖、批量修改、提权等危险操作加保护措施
- `MUST` 避免未引用变量、未转义参数、危险通配、未受控子命令执行
- `SHOULD` Shell 脚本遵循 `Google Shell Style Guide` 并通过 `ShellCheck`
- `SHOULD` PowerShell 脚本遵循 `PSScriptAnalyzer`

#### 5.2.3 可靠性与可恢复性

- `MUST` 明确失败退出行为，避免半完成状态静默继续
- `SHOULD` 对高风险脚本提供 `dry-run`、确认开关、备份或回滚说明
- `SHOULD` 对批量操作输出最小必要的进度和失败摘要

#### 5.2.4 备注与可读性

- `MUST` 仅保留必要备注，重点说明危险点、前置条件、恢复方式
- `MUST NOT` 用大量注释替代清晰结构与安全写法

#### 5.2.5 验证

- `MUST` 提供至少一种可执行验证：语法检查、静态分析、样例运行、dry-run、退出码验证
- `MUST` 对高风险脚本提供替代验证证据，即使不能在当前环境真实执行

### 5.3 文档（Documentation）

#### 5.3.1 通用要求

- `MUST` 先明确文档类型、目标受众、使用场景
- `MUST` 使用清晰结构，避免把教程、参考、设计决策、运维步骤混写
- `MUST` 保证术语一致、步骤顺序正确、内容与实际实现/流程一致
- `MUST` 删除或标记过时信息，避免读者被旧信息误导
- `SHOULD` 采用 `Diataxis` 的文档分层思路，并参考 `Google` / `Microsoft` 写作风格
- `SHOULD` 对关键文档标注维护责任与更新时间

#### 5.3.2 需求文档 / PRD

- `MUST` 包含目标、范围、非目标、关键场景、约束、验收标准
- `SHOULD` 参考 `ISO/IEC/IEEE 29148` 的需求清晰性、一致性、可验证性原则

#### 5.3.3 设计文档 / ADR

- `MUST` 包含上下文、决策、备选方案、权衡、影响范围、失败模式或回滚考虑
- `SHOULD` 说明为何选当前方案而不是其他方案

#### 5.3.4 测试文档 / 测试用例

- `MUST` 包含范围、前置条件、步骤、输入、预期结果、环境或数据假设
- `SHOULD` 参考 `ISO/IEC/IEEE 29119-3` 的测试文档工作产品结构

#### 5.3.5 运维文档 / Runbook / SOP

- `MUST` 包含触发条件、前置条件、操作步骤、验证方式、回滚或升级路径
- `MUST` 保证命令、路径、依赖、权限要求可执行
- `SHOULD` 标注风险等级与人工升级条件

#### 5.3.6 API / 开发者文档

- `MUST` 包含接口用途、参数/字段、返回值、错误、版本或兼容性说明
- `SHOULD` 提供最小可运行示例与常见错误排查

#### 5.3.7 教程 / 用户指南 / Reference

- `MUST` 区分前置条件、操作步骤、预期结果、常见问题
- `SHOULD` 避免把概念解释、逐步操作、API 参考混在同一层级

## 6. 验证顺序

按适用性执行，不得只挑最省事的验证项：

1. 识别产物类型与适用门禁
2. 执行基础校验
   - 代码：`Typecheck / Build`
   - 脚本：语法检查 / 静态分析
   - 文档：结构回读 / 示例与命令回读
3. 执行质量校验
   - 代码：`Lint / Static Analysis`
   - 脚本：`ShellCheck` / `PSScriptAnalyzer` / 等价工具
   - 文档：链接、术语、一致性、命令可执行性检查（如可行）
4. 执行行为验证
   - 代码：自动化测试 / 手动验证
   - 脚本：样例执行 / dry-run / 退出码验证
   - 文档：按步骤走读或抽样执行
5. 汇报验证证据与受限项

纯文档任务可跳过编译与自动化测试，但不得跳过回读与一致性检查。

## 7. Review 自检与低成本优化

每轮实现或编写结束后，必须执行一次最小自检：

1. 先识别 `阻断项` 与 `低成本优化项`
2. `MUST` 先修复阻断项
3. `SHOULD` 默认修复触及范围内的低成本优化项：
   - 死代码
   - 冗余或过时注释
   - 局部低效实现
   - 明显可读性噪音
   - 文档中的错误标题、错别字、失效引用、结构噪音
4. 若优化会引入较高行为风险，必须标记为 `defer` 并说明原因
5. 修复后必须按第 6 节重新执行适用验证
6. 同一问题最多迭代 `3` 轮；仍未收敛时必须披露残余风险

## 8. 立即失败（Immediate Fail）红旗

出现以下任一情况直接判定为 `FAIL`：

- 未完成目标却声称“已完成”
- 已知关键错误、关键缺口、关键限制被隐藏或弱化
- 引入或保留高危漏洞：注入、越权、硬编码凭证、敏感信息泄漏、不安全动态执行
- 危险脚本缺少最小保护，可能造成不可控破坏
- 运维文档、SOP、runbook 中关键步骤不可执行或明显误导
- 没有验证证据却声称“已验证”“已通过”
- 使用 `waiver` 掩盖阻断级安全或正确性问题

## 9. 标准来源与设计依据

本规则参考但不限于以下业界基线：

- `ISO/IEC 25010`：系统与软件质量模型
- `NIST SP 800-218 SSDF`：安全软件开发框架
- `OWASP ASVS`：应用安全验证要求
- `OWASP Top 10`：高频应用安全风险
- `OWASP SAMM`：软件保障成熟度模型
- `MITRE CWE Top 25`：高危弱点类别
- `CISA Secure by Design`：secure-by-default / secure-by-design 原则
- `SLSA`：软件供应链完整性与来源保障
- `ISO/IEC/IEEE 29148`：需求工程与需求质量
- `ISO/IEC/IEEE 29119-3`：测试文档工作产品
- `Diataxis`：文档类型分层
- `Google Developer Documentation Style Guide`
- `Microsoft Writing Style Guide`
- `Google Shell Style Guide`
- `ShellCheck`
- `PSScriptAnalyzer`

本规则不是这些标准的逐字转录，而是将其提炼为 AI agents 可执行、可判定、可汇报的统一质量门禁。

## 10. 快速判定表

### 10.1 PASS

- 所有适用 `MUST` 满足
- 无阻断项
- 验证证据完整
- 风险已充分披露

### 10.2 CONDITIONAL PASS

- 所有适用 `MUST` 满足
- 存在非阻断缺口
- 风险、补救动作、后续计划已明确

### 10.3 FAIL

- 任一适用 `MUST` 未满足
- 存在阻断项
- 证据缺失或结论与证据不一致
